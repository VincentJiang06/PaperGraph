# Loop runbook — Implement PaperGraph v1 (src/paperproof + tests) from docs/00–12, milestones M0–M4 with docs/11 §9 gates, executed by Claude Code (Opus); live smoke stays a manual human node.

> A runnable closed-loop design generated by loop-constructor. Follow it as a loop: make a small change → run the stage check → read the result → fix → repeat until the check passes, then advance.
> **Altitude:** medium — Blast radius is repo-local code (no prod surface), fully git-reversible, and docs/10 §7 mandates strictly sequential milestones — so sequential gated stages with a single generator; fan-out inside a stage is an implementation detail, not loop structure.

## Hard constraints
- Minimal viable change; don't touch unrelated files.
- Don't change public APIs / add deps without justification.
- No production resources.
- On a requirement contradiction → STOP and report.

## Roles (three separate contexts — never one agent grading itself)
- **Planner:** Reads docs/00–12 + AGENTS.md/CLAUDE.md only. At each stage entry, emits the stage plan (modules, file list, build order, doc sections cited) into .loop/state/progress.md and keeps .loop/state/contract.md current. Never writes src/ or tests/ code.
- **Generator:** Claude Code (Opus). Implements src/paperproof and tests for the current stage per the plan and the cited docs; runs the gate command and records the report. Forbidden from: declaring its own gate final (the evaluator does), weakening or deleting fixtures/tests to pass, changing docs/ to legalize a bug without flagging the doc-change in progress.md, or adding CLI/schema surface beyond the closed lists (docs/10 §4, docs/08) — that requires escalation.
- **Evaluator (fresh context, adversarial — told the artifact is broken, proves it):** Fresh context per gate that never saw the generator's reasoning: receives only docs/, the stage diff vs the previous gate tag, and the gate report. Told the milestone is broken and must prove it: re-runs the gate command, re-derives sampled rules from docs/09 and audits the fixtures, hunts weakened tests (diff vs gate tag + untracked scan), and for m2/m3/m4 executes its named adversarial probe (own fixture / mangled annotations / delete db mid-test). Writes PASS or FAIL+findings to the gate report; only its PASS closes a gate.

## Why this shape (decision log)
> The D0–D6 selection procedure that derived this loop's shape.
- **D0** (loop) — Done is machine-decidable per milestone: `pytest -q` green (default markers) + required-file presence, and at the end a fresh-clone suite — docs/11 §9 defines exactly these gates; no human needs to read output to close a stage.
- **D1** (staged (5 seams)) — M0→M1→M2→M3→M4→final-audit: each milestone hands a stable artifact the next consumes unchanged (schemas/store → proof loop → docs loop → endgame → surface → auditable package), each upstream gate runs without downstream code existing (CLI stubs from M0 keep the conformance meta-test green), and milestones own disjoint modules.
- **D2** (plan_execute_verify for m0–m4, review for final-audit) — Each milestone is a large, ambiguous build against a dense spec — needs mid-flight self-correction, not blind retry; the final stage IS a correctness judgment (fresh-clone + adversarial audit). Checks are the cumulative pytest suite + file-presence compounds that fail on missing or broken work.
- **D3** (on_the_loop) — Low blast radius (local repo), high reversibility (git), strong feedback (deterministic suite + meta-tests that police their own coverage) — human reviews at gates and owns only live smoke, contract disputes, and doc changes.
- **D4** (medium) — docs/10 §7: no milestone starts before the previous acceptance — the stage graph is a chain with zero independent stages, so multi-agent fan-out buys nothing at loop level.
- **D5** (caps 3/4/3/3/3/2, restart for m1–m4, escalate for m0 and final-audit, outer budget 3) — m1 (committer/queue/decision table) is where patch-piles rot into archaeology → restart re-derives from the contract; m0 failing 3 passes implicates the spec itself → escalate; final-audit failures beyond 2 mean environment/contract issues → escalate; reward-hacking, spec-drift, token-blowup guards below.
- **D6** (completeness_first) — Iteration boundaries are expensive (each pass must reload a 3k-line spec) and the check is thorough-but-slow (full cumulative pytest + verify), so few long passes: low caps (≤4), plan_execute_verify, whole-milestone scope per pass, full-suite gates instead of smoke checks.

## Stages (run in dependency order; each stage is its own gated loop)
> After a stage gate goes green, commit/checkpoint it; on a loopback, reset to the last green checkpoint rather than editing forward.

### Stage 1 — `m0-foundation`  ·  pattern: plan_execute_verify

- **Done when:** Gate command exits 0: full `pytest -q` green AND every docs/11 §9 M0-required test file exists (schemas, textutil, ids, clock, store, snapshot, scoping, V-SPEC/V-PATH rules, project/spec CLI + full stub surface registered).
- **Check (run — the gate):** `pytest -q && test -f tests/unit/test_textutil.py && test -f tests/unit/test_jsonl_store.py && test -f tests/unit/test_snapshot.py && test -f tests/unit/test_ids.py && test -f tests/contract/test_schemas.py && test -f tests/contract/test_v_spec.py && test -f tests/contract/test_v_path.py && test -f tests/contract/test_cli_envelope.py`  ·  expect: `pass`
- **Falsifiable when:** The topic parser mishandles the P4 example's all-`#` headings (docs/01 P1) so the byte-exact golden PaperSpec test fails; or the CLI conformance test finds a command missing from the registered stub surface; or an unknown-field schema fixture is accepted.
- **Passing-but-wrong (ruled out):** Generator creates the required test files with trivial/empty assertions so pytest greens on hollow tests — the file-presence check cannot see assertion quality; caught by the evaluator's fixture audit (re-deriving sampled V-SPEC rules from docs/09 and checking the fixtures encode them) plus the rule-coverage meta-test once it lands in m1.
- **Depends on:** — (entry stage)  ·  admit downstream only once this check passes.
- **Stop:** max 3 iterations; failure branches:
    - pytest red after a full pass
    - a required M0 test file is missing
    - gate not green within the cap
- **On failure:** escalate.

### Stage 2 — `m1-proof-loop`  ·  pattern: plan_execute_verify

- **Done when:** Gate command exits 0: full `pytest -q` green AND all M1-required files exist (decision table 24 goldens + totality fuzz, V-PR/V-EXP/V-TASK/V-Q/V-COMMIT fixtures, rule-coverage meta-test, scenarios S1/S4/S5/S6, determinism byte-identity).
- **Check (run — the gate):** `pytest -q && test -f tests/unit/test_decision_table.py && test -f tests/contract/test_v_pr.py && test -f tests/contract/test_v_exp.py && test -f tests/contract/test_v_task.py && test -f tests/contract/test_v_q.py && test -f tests/contract/test_v_commit.py && test -f tests/contract/test_rule_coverage.py && test -f tests/integration/test_s1_seed_loop.py && test -f tests/integration/test_s4_parallel.py && test -f tests/integration/test_s5_crash.py && test -f tests/integration/test_s6_stale.py && test -f tests/integration/test_determinism.py`  ·  expect: `pass`
- **Falsifiable when:** Decision table returns pass for golden row N01 (out_of_scope must reject); S1 finds no wired bridge edges C→B/D→B after the needs_repair(bridge) commit; two identical S1 runs differ by one byte in graph/logic_nodes.jsonl; a hostile H-fixture is accepted without its named V-rule in failed_rules.
- **Passing-but-wrong (ruled out):** Determinism test compares parsed JSON objects instead of raw bytes, hiding canonical-serialization bugs; or the totality fuzz iterates a hand-copied enum list that omits not_evaluated, vacuously passing — evaluator re-derives the enum product size from docs/03 and checks the fuzz case count against it.
- **Depends on:** `m0-foundation`  ·  admit downstream only once this check passes.
- **Stop:** max 4 iterations; failure branches:
    - pytest red after a full pass
    - same assertion failing on two consecutive attempts without a written diagnosis in progress.md
    - gate not green within the cap
- **On failure:** restart.

### Stage 3 — `m2-docs`  ·  pattern: plan_execute_verify

- **Done when:** Gate command exits 0: full `pytest -q` green AND M2-required files exist (V-DR fixtures incl. quote_match hostile, S2 cache/cap loop, S3 cascade).
- **Check (run — the gate):** `pytest -q && test -f tests/contract/test_v_dr.py && test -f tests/integration/test_s2_docs_loop.py && test -f tests/integration/test_s3_contradiction.py`  ·  expect: `pass`
- **Falsifiable when:** S2's identical second request dispatches a docs work item instead of resolving fulfilled_by="cache"; the third needs_docs verdict fails to produce a born-dead item; hostile D02 (fake quote) passes quote_match; S3 cascade leaves an incident edge non-rejected.
- **Passing-but-wrong (ruled out):** The matcher is tuned to the exact fixture tokens (overfit to the two-source mini corpus) and would miss on any real corpus — the suite cannot see retrieval quality beyond its fixtures; evaluator adds one adversarial fixture of its own devising at the gate.
- **Depends on:** `m1-proof-loop`  ·  admit downstream only once this check passes.
- **Stop:** max 3 iterations; failure branches:
    - pytest red after a full pass
    - gate not green within the cap
- **On failure:** restart.

### Stage 4 — `m3-endgame`  ·  pattern: plan_execute_verify

- **Done when:** Gate command exits 0: full `pytest -q` green AND M3-required files exist (V-FRZ/V-CDR/V-PROSE-AUD fixtures, S7 full pipeline incl. MSA-1..9 + zero-gap dry run + trace, S1 freeze coda + S3 MSA-9 coda).
- **Check (run — the gate):** `pytest -q && test -f tests/contract/test_v_frz.py && test -f tests/contract/test_v_cdr.py && test -f tests/contract/test_v_prose_aud.py && test -f tests/integration/test_s7_full_pipeline.py`  ·  expect: `pass`
- **Falsifiable when:** msa-check passes on the S3-coda fixture whose thesis support was hollowed out (MSA-9 must fail it); the post-freeze dry run reports a gap on a clean freeze (docs/06 says zero by construction); trace fails to resolve a spine node down to the raw file; a V-FRZ precondition violation is not refused.
- **Passing-but-wrong (ruled out):** S7's prose fixtures are authored to trivially satisfy V-PROSE (every sentence annotated, no unannotated transitions), so the annotation-grammar parser is never exercised on realistic prose — evaluator feeds the seeded forbidden-language variant and one hand-mangled annotation file at the gate.
- **Depends on:** `m2-docs`  ·  admit downstream only once this check passes.
- **Stop:** max 3 iterations; failure branches:
    - pytest red after a full pass
    - gate not green within the cap
- **On failure:** restart.

### Stage 5 — `m4-surface`  ·  pattern: plan_execute_verify

- **Done when:** Gate command exits 0: full `pytest -q` green AND M4-required files exist (S8 rebuild/corruption, /api endpoint tests answering the six Overview questions; ui static page present).
- **Check (run — the gate):** `pytest -q && test -f tests/integration/test_s8_rebuild.py && test -f tests/integration/test_api.py && test -f src/paperproof/ui/static/index.html`  ·  expect: `pass`
- **Falsifiable when:** A corrupted JSONL line is silently skipped instead of exiting 3 with file+line; db rebuild twice yields different index_manifest hashes; /api/overview cannot answer one of the six questions from the S7-shaped fixture; the stale_index flag stays false after touching a source JSONL.
- **Passing-but-wrong (ruled out):** Endpoints return values computed from the JSONL directly instead of the DuckDB index, making the stale-banner mechanism decorative while all read-tests pass — evaluator deletes db/ mid-test to prove reads actually go through the index.
- **Depends on:** `m3-endgame`  ·  admit downstream only once this check passes.
- **Stop:** max 3 iterations; failure branches:
    - pytest red after a full pass
    - gate not green within the cap
- **On failure:** restart.

### Stage 6 — `final-audit`  ·  pattern: review

- **Done when:** Fresh-clone gate exits 0: clone into a mktemp dir, venv-install `.[dev]`, run the full default suite there — proving the package carries no dev-environment or repo-path dependence.
- **Check (run — the gate):** `AUDIT=$(mktemp -d) && git clone -q . "$AUDIT/repo" && python3 -m venv "$AUDIT/venv" && "$AUDIT/venv/bin/pip" -q install -e "$AUDIT/repo[dev]" && cd "$AUDIT/repo" && "$AUDIT/venv/bin/python" -m pytest -q`  ·  expect: `pass`
- **Falsifiable when:** The suite passes in the dev checkout but fails from a fresh clone (uncommitted fixture, absolute path, missing package-data entry for prompts/ or ui/static/, undeclared dependency).
- **Passing-but-wrong (ruled out):** A green fresh-clone suite still cannot prove spec conformance beyond what the tests encode — the adversarial evaluator pass over the contract (sampled rule re-derivation, weakened-test hunt vs gate tags) is the explicit residue-catcher, and live smoke with real workers stays a human node by design.
- **Depends on:** `m4-surface`  ·  admit downstream only once this check passes.
- **Stop:** max 2 iterations; failure branches:
    - fresh-clone suite red
    - evaluator files a hollow-gate finding the generator cannot refute
    - gate not green within the cap
- **On failure:** escalate.

## Stop conditions (overall)
- **Success:** Stages m0-foundation through final-audit all gate-green at HEAD: full `pytest -q` passes (default markers), every machine assertion A1–A25 holds, the fresh-clone suite passes, and each gate carries an evaluator PASS verdict. Live smoke (A26) and doc-discipline spot-audit (A27) remain deliberate human nodes and are recorded, not automated.
- **Max iterations (budget cap):** 3
- **Failure branches:**
  - a stage exhausts its inner cap without a green gate
  - the outer budget of 3 full-loop re-entries is exhausted
  - the evaluator proves the same stage's gate green-but-hollow twice
- **Escalate to a human when:**
  - a contract assertion turns out wrong or unsatisfiable against docs/ — a spec bug; humans own the doc change
  - the build needs a command or schema outside the closed surfaces (docs/10 §4, docs/08) — the doc must be amended first, a human decision
  - the evaluator finds deliberately weakened or deleted fixtures/tests
  - live smoke reveals a worker-protocol spec failure rather than a code bug

## Terminal states
- **SUCCESS** — Stages m0-foundation through final-audit all gate-green at HEAD: full `pytest -q` passes (default markers), every machine assertion A1–A25 holds, the fresh-clone suite passes, and each gate carries an evaluator PASS verdict. Live smoke (A26) and doc-discipline spot-audit (A27) remain deliberate human nodes and are recorded, not automated. → done.
- **STOPPED_UNMET** — a max-iteration cap is hit or a failure branch fires: stop, report the last green stage and the unmet gate, hand back the partial diff, do NOT merge. Never relax a gate to force a pass.
- **ESCALATE** — one of the escalate triggers above fires: hand to a human.

## Human placement
- on_the_loop

## Checker (separate, fresh-context)
- Per gate: adversarially review the stage diff (git diff gate/&lt;prev&gt; -- src tests) plus the gate report against the contract — re-derive 3 randomly sampled V-rules from docs/09 and confirm the fixtures encode them, hunt weakened/deleted tests vs the previous gate tag (including untracked files via git status --porcelain), and re-run the gate command in a clean shell.

## Risk guards
- **Reward hacking: the generator weakens fixtures, deletes hostile cases, or skips tests to green a gate** → Evaluator diffs tests/ + fixtures/ against the previous gate tag including untracked files; the rule-coverage meta-test makes fixture deletion loud (a rule without fixtures fails the build); weakening found twice ⇒ escalate
- **Spec drift: implementing from memory of the docs instead of the docs** → Stage plans must cite the doc sections they implement; the evaluator re-derives 3 sampled V-rules from docs/09 at every gate and compares against the fixtures; CLAUDE.md same-commit doc-sync rule spot-audited (A27)
- **Archaeology: patching the committer/queue until the code is a patch-pile** → on_failure=restart for m1–m4 discards the stage's work and re-derives from the contract; restarts are autonomous — no human interrupt (humans are reserved for wrong contracts)
- **Context drift / compaction loss across a multi-day build** → All loop state lives in .loop/state/* (contract, progress, log, gate reports); resume = re-read those files, never trust a session summary
- **Token blowup: unbounded retries against a red suite** → Completeness-first caps (≤4 per stage, 3 outer); a same-assertion second failure requires a written diagnosis in progress.md before the next attempt
- **Dev-environment dependence: suite green only in the working checkout** → final-audit gate clones into mktemp + clean venv and re-runs the whole suite there (A25)

## Harness primitives
- .loop/state/contract.md — the negotiated assertions A1–A27 (the graded criteria), rendered from this design
- .loop/state/progress.md — per-stage status, current plan, next actions, failure diagnoses; the resume point after any compaction or crash
- .loop/state/log.md — append-only, one line per operation: ## [date] op | title
- .loop/state/gate-reports/&lt;stage&gt;-attempt&lt;N&gt;.md — gate command tail + evaluator verdict per attempt
- git tag gate/&lt;stage&gt; at each green gate — the pinned baseline the evaluator diffs tests/ and fixtures/ against (git diff gate/&lt;prev&gt; -- tests; git status --porcelain for untracked)
- .loop/state/traces/ — captured generator run transcripts for read-the-traces debugging
