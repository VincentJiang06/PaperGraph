# PaperGraph 产品形态与使用方式预测

> 本文是产品规划，不是构建规范。实现以 `docs/` 为准；本文预测的是 v1 建成之后，这个东西在真实世界里长什么样、被谁、怎么用。

## 一句话定义

**PaperGraph 是一台"论证编译器"：先把论文的论证结构建成可证明的逻辑图，每条推理边都被证据绑定并通过判定表裁决，最后才一次性编译成散文——并附带完整的审计链。**

它站在所有"AI 写作工具"的反面：别人帮你更快地产出文本，它阻止你在论证站住之前产出任何文本。

## 产品本体：三层结构

```text
第一层  规范（docs/ 00–12）     —— 可移植的方法论标准。任何 agent harness
                                  理论上都能实现它；这是产品的灵魂。
第二层  paperproof（Python 包） —— 确定性的账房：schema、队列、判定表、
                                  提交、冻结、编译、审计、WebUI。pip 可装。
第三层  编排工作流              —— 跑在 Claude Code 里的 Orchestrator 协议
                                  （CLAUDE.md + prompts/），驱动 subagent
                                  worker 干判断性工作。
```

用户拿到手的"产品"= 一个模板仓库：clone 下来，`pip install paperproof`，在 Claude Code 里打开，说"开始一篇论文"，剩下的由 CLAUDE.md 协议接管。

## 形态演进预测

```text
v1   (现规范)   Claude Code 高级用户工具。本地文件、单项目、单模式
                （single_event_mechanism）、localhost WebUI。
                形态：模板仓库 + PyPI 包。受众：极客型研究者。

v1.5 (自然演化) 一键化。`paperproof new` 脚手架 + grill-me 选题访谈 +
                更多论文模式模板 + BINDING_CHECK + 语义审计 worker。
                形态：开箱即用的 CLI 产品。受众：不想读文档的研究生。

v2   (分叉点)   两条路二选一或并行：
                a) 协作层——导师/合作者通过只读链接审阅冻结的 spine，
                   评论落在节点上而非文档段落上（"审图不审稿"）；
                b) 反向编译——导入已有草稿，抽取声明建图、逐边验证，
                   变成"论证审计器"。b 的市场比 a 大一个数量级：
                   存量论文远多于增量论文。
```

## 目标用户画像（按适配度排序）

```text
1. 人文社科/经济/法律/政策方向的博士生与青年学者
   痛点：论证型长文写到一半发现结构塌了；导师反馈"逻辑不通"但说不清哪不通。
   卖点：Logic Map 就是答辩地图——每条边被质疑时，trace 直达证据原文。

2. 政策分析师 / 智库 / 咨询
   痛点：报告的每个结论都可能被客户/上级追问出处；引用错误是职业风险。
   卖点：审计链（sentence → evidence → raw file）是合规资产，不是学术情怀。

3. 深度报道记者 / 长文作者
   痛点：事实核查成本高；核查与写作割裂。
   卖点：Docs Database 的 can/cannot_cite_for 就是内置核查员。

4. 独立研究者 / 写严肃 newsletter 的人
   痛点：没有同行评审环境，论证质量靠自觉。
   卖点：Proof Machine 是一个不知疲倦的抬杠者。

5. 高校写作中心 / 研究方法课教师（渠道型用户）
   用法：把"看着自己的论证被拆掉再修好"作为教学工具。
```

**反用户**：赶 deadline 要"三小时出一篇"的人。PaperGraph 会让他们更慢——这是特性不是缺陷，但要在宣传里讲清楚，否则差评来自错配预期。

## 典型使用旅程（主模式，v1）

```text
第 0 步  准备（30 分钟，人）
  写 topic-input.md：核心问题、论点方向、范围、排除项、种子声明。
  这一步的认知投入是整个方法的第一道价值——多数人从没被迫想清楚过。

第 1 步  定界（10 分钟，人机对话）
  paperproof spec build → Claude 逐条追问范围与排除项 → 人确认 spec accept。
  合同锁定后，人的角色从"作者"切换为"监工"。

第 2 步  建图与证明（2–6 小时，机器为主，人偶尔介入）
  层 0 播种 → 并行 ProofWorker 填表 → 判定表裁决 → 桥接/找文献/收窄循环。
  人打开 localhost:8420 看 Logic Map 从蓝变绿；被叫来的时刻只有：
  dead letter 仲裁、alternative 的 park 决定、意外的范围争议。

第 3 步  冻结与编译（30 分钟）
  msa-check 绿 → spine freeze → dry run 补 gap → CompileWorker 逐节写作
  → 机械审计零发现。

第 4 步  收获（人）
  带注解的散文（去掉注解即成稿）+ 逻辑图导出（答辩附件）+
  证据附录（自动引用表）+ trace 报告（给审稿人的可复现包）。
```

其他模式：**文献先行**（先把一文件夹 PDF 灌进 docs DB 再开题）、**只建图不写作**（开题报告/研究设计阶段，产出就是图本身）、**审阅协作**（v2）。

## 成本与耗时的现实预测（要在 README 里诚实写出）

```text
一篇 30–50 节点、60–100 个证明任务的论文规模项目：
  token 量  ~1M–3M（ProofWorker 每任务 3–8K 入 1–2K 出；DocsWorker 更重）
  钱        Sonnet 级 worker 约 $5–25/篇；全 Opus 约 3–5 倍
  墙上时间  半天到两天（含人介入的间隙）
  对比锚点  一次靠谱的人工文献核查外包 > $200；一次论证返工 = 数周
```

## 差异化定位

```text
Elicit / Consensus     文献发现层——它们找证据，PaperGraph 消费证据并绑定到论证
NotebookLM / RAG chat  问答层——聊天不留下可审计结构
Jenni / 各类 AI 写作    文本生产层——正面竞品，也是定位上的"反面教材"
Scite                  引用核查层——单点功能，PaperGraph 把它内建为 V-DR 规则
知网/Zotero 生态        管理层——可做上游数据源集成

空位：没有任何产品占据"论证结构的可验证性"。这是类目创造机会
（category creation），也是最大的教育成本所在。
```

## 风险与摩擦点预测

```text
1. 纪律成本：合同先行、无散文中间态，违反所有人的写作直觉。
   → 用 5 分钟的"看图变绿"演示视频降低想象成本；反向编译（v2b）作后门。
2. Claude Code 依赖：harness 换代或定价变化直接冲击。
   → 规范层刻意 harness 无关；worker 协议只是"读文件、写文件"。
3. 学术界对 AI 的污名：会被误读为代写工具。
   → 定位死咬"验证器而非生成器"；prose 是最后 5%，且全程留审计痕迹——
     它产出的恰恰是"用了 AI 但敢给你看全过程"的论文。
4. 非英文文献的抽取质量（中文 PDF 尤其）。
   → textutil 已支持 CJK；PDF 抽取质量要在 v1.5 重点投入。
5. 真实论证比 P4 示例脏得多：循环论证、隐含前提、规范性声明。
   → bridge/narrow 机制吃掉一部分；剩下的靠 dead letter 升级给人——
     要把"系统知道自己不知道"作为卖点讲。
```
